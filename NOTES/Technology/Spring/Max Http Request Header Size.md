# Max Http Request Header Size

요청 헤더가 너무 크면 애플리케이션 로직까지 오기 전에 서버가 요청을 거절할 수 있다. 증상은 단순하지만, 원인은 보통 비즈니스 코드보다 웹 서버, 인증 구조, 프록시 체인 쪽에 있다.

## 한눈에 보기
- 대표 증상: `400`, `431`, container-level request rejection
- 흔한 원인: 긴 쿠키, 과도한 JWT, 프록시 헤더 누적
- 조정 지점: Spring Boot의 내장 서버 설정
- 우선순위: 한도 상향보다 원인 축소를 먼저 검토

## 왜 보게 되나

- 쿠키가 비대해짐
- Authorization header가 너무 큼
- 프록시/게이트웨이에서 헤더가 계속 추가됨

## 핵심 관점

- 이건 비즈니스 코드 문제가 아니라 **웹 서버/컨테이너 제한** 문제다
- Spring Boot에서는 내장 서버 설정과 연계해서 조정한다
- 무조건 크게 열기보다 왜 헤더가 커졌는지 먼저 본다

Baeldung 정리처럼 버전에 따라 속성 키가 달라질 수 있고, 실제 한도는 사용하는 서버 구현체(Tomcat, Jetty, Undertow)에도 영향을 받는다. 그래서 "Spring Boot 설정 한 줄"만으로 끝나는 문제가 아닐 때가 있다.

## 실무 포인트

- 세션/쿠키 구조가 비대하지 않은지 본다
- JWT payload를 과도하게 넣고 있지 않은지 점검한다
- 프록시 체인을 거치며 헤더가 불필요하게 누적되는지 확인한다

## 조정이 필요할 때도 보는 순서가 있다
1. 어떤 헤더가 커졌는지 확인
2. 애플리케이션 서버 한도 확인
3. 프록시, 게이트웨이, LB 한도도 함께 확인

즉 내장 Tomcat 설정만 늘리고 끝낼 수 있는 문제가 아닐 수 있다.

## 왜 설정만으로 끝내면 안 되나
한도를 늘리면 당장 증상은 사라질 수 있지만, 헤더가 비대해진 원인 자체는 남는다. 특히 쿠키 설계나 인증 토큰 구조가 커지는 문제는 나중에 또 다른 운영 이슈로 이어질 수 있다.

예를 들어 access token 하나에 과도한 클레임을 넣고, 여기에 여러 쿠키가 더해진다면 서버 한도만 키우는 방식은 금방 다시 한계에 부딪힌다.

## 정리

`Max Http Request Header Size`는 설정값 문제처럼 보이지만, 실제로는 인증/쿠키/프록시 구조를 다시 보라는 신호인 경우가 많다. 한도를 늘릴 수는 있어도, 헤더가 커지는 이유를 먼저 줄이는 편이 운영적으로 더 안전하다.

즉 "서버 설정을 올린다"보다 "어떤 헤더가 왜 커졌는지 파악한다"가 먼저다.
