# Java `final` Keyword

`final`은 한 가지 의미가 아니라 대상에 따라 역할이 다르다.

- 변수: 재할당 금지
- 메서드: 오버라이드 금지
- 클래스: 상속 금지

참조형 변수에 붙은 `final`은 참조 재할당을 막을 뿐, 객체 내부 상태 변경까지 막지는 않는다는 점을 자주 헷갈린다.

## 변수에 붙을 때

```java
final List<String> names = new ArrayList<>();
names.add("kim");          // 가능
names = new ArrayList<>(); // 컴파일 오류
```

즉 `final`은 "이 변수가 가리키는 참조를 바꾸지 않는다"는 뜻이지, 리스트를 불변으로 만든다는 뜻이 아니다.

## 메서드와 클래스에 붙을 때
메서드에 붙으면 하위 클래스 오버라이드를 막고, 클래스에 붙으면 상속 자체를 막는다. 그래서 `final`은 단순 문법이 아니라 "여기는 확장 지점이 아니다"라는 설계 메시지를 주는 수단이 되기도 한다.

## 왜 중요한가
`final`은 단순 제약이 아니라 의도를 드러내는 문법이다. "이 참조는 바뀌지 않는다", "이 메서드는 재정의되면 안 된다", "이 타입은 확장 지점이 아니다" 같은 설계 메시지를 함께 전달한다.

## 자주 헷갈리는 지점
참조형 변수에 `final`을 붙였다고 객체가 불변이 되는 건 아니다. 불변성과 재할당 금지는 다른 개념이다. 이 차이를 모르고 `final`을 남용하면 오히려 잘못된 안정감을 줄 수 있다.

## 불변성과 연결해서 볼 때
진짜 불변 객체를 만들려면 보통 다음 조건이 함께 필요하다.

- 필드 참조를 `final`로 고정
- 내부 상태 변경 메서드를 노출하지 않음
- 가변 필드를 외부에 그대로 노출하지 않음

즉 `final`은 불변 설계의 한 요소일 수는 있어도, 그것만으로 불변을 완성하지는 못한다.

## 실무 기준
- 로컬 변수/필드: 재할당 금지 의도가 있으면 적극 사용
- 클래스/메서드: 상속 확장 정책이 명확할 때만 사용
- 불변 객체 설계: `final`만으로 끝내지 말고 내부 상태도 함께 통제

## 정리
`final`은 문법 기능이면서 동시에 설계 의도를 드러내는 도구다. 다만 불변성과 같은 개념으로 혼동하면 안 된다.
