# Java Varargs

varargs는 인자 개수가 가변적일 수 있다는 의도를 API에 바로 드러내는 문법이다. 호출부는 간단해지지만, 내부적으로는 결국 배열로 처리된다는 점을 같이 이해해야 한다.

## 기본 예제

```java
static int sum(int... values) {
    int result = 0;
    for (int value : values) {
        result += value;
    }
    return result;
}
```

호출부에서는 `sum(1, 2, 3)`처럼 자연스럽게 쓸 수 있지만, 메서드 안에서는 사실상 `int[]`를 다루는 것과 같다.

## 어디에 잘 맞나
- 로그 유틸리티
- 간단한 합성 함수
- 옵션 값 목록

즉 여러 개를 받되, 인자 수 자체가 API 의미의 일부인 경우에 잘 맞는다.

## 배열이라는 사실을 잊으면 안 된다
varargs는 문법상 편해 보여도 배열 생성 비용이 완전히 사라지는 것은 아니다. 그래서 아주 자주 호출되는 hot path에서는 무심코 쓰기보다, 실제로 가변 인자가 필요한지 다시 보는 편이 좋다.

## 오버로딩과 섞일 때 주의
과도한 오버로딩과 결합하면 어떤 메서드가 호출되는지 애매해질 수 있다. 특히 `String...`, `Object...`, 일반 배열 버전이 함께 있을 때 가독성과 예측 가능성이 빠르게 떨어진다.

## 마지막 파라미터에만 둘 수 있다는 점도 의미가 있다
varargs는 문법적으로 마지막 인자에만 둘 수 있다. 그래서 API를 설계할 때 "고정 인자 + 가변 인자" 구조로 읽히는 경우에 특히 자연스럽다.

## 실무 기준
- 호출부 편의성이 중요한 유틸 API: varargs 적합
- 성능 민감한 반복 경로: 배열 생성 비용 점검
- 오버로딩이 많다면 varargs 남용 지양

또한 가변 인자가 너무 많은 메서드는 호출은 편해 보여도 의미가 흐려질 수 있다. 인자 의미가 복잡해지면 전용 객체로 묶는 편이 더 낫다.

## 정리
varargs는 편의 문법이지만 결국 배열 기반이다. API 의도를 분명하게 만들 때만 쓰는 편이 좋다.
