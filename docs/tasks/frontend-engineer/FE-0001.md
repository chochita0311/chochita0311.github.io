# FE-0001: MVP Board Search and Post Detail UI

## Owner
- Agent: FE (frontend_engineer)

## Status

## Goal
- Build PM-0001 frontend MVP in static mode:
  - left tab `My Notes`
  - searchable post list
  - post detail view
  - cafe-board style form (left menu + dense table-like list)
  - local data mode using separated table files in `docs/tables/localdb/`

## Scope
- In:
  - left navigation with one tab (`My Notes`)
  - search input + result list UI
  - table-like list row layout with high information density
  - post detail view UI
  - URL-based navigation between list/detail
  - state handling (`default`, `loading`, `empty`, `error`, `success`)
  - mobile-first responsive behavior
  - data adapter boundary for future BE API switch
- Out:
  - post create/edit/delete UI implementation
  - auth/login
  - reservation/calendar
  - board/community feature UI expansion
  - backend API implementation

## FE API Contract Draft (PM-0003 Material)

### 1) List/Search Posts
- Method: `GET`
- Path: `/api/posts`
- Query params:
  - `boardId`: number (required)
  - `q`: string (optional)
  - `folder`: string (optional)
  - `limit`: number
  - `offset`: number

### 2) Post Detail
- Method: `GET`
- Path: `/api/posts/{post_id}`

### 3) Create Post (deferred)
- Method: `POST`
- Path: `/api/posts`
- Request body:
```json
{
  "board_id": 1,
  "title": "New post",
  "body_md": "content"
}
```

## Acceptance Criteria
- User can browse `My Notes`, search posts, and open detail from result list.
- Back navigation returns user to previous search context.
- Core interactions are usable on desktop and mobile.
- UI states are clearly rendered for loading/empty/error.
- FE runs without `notes.json`; it reads `docs/tables/localdb/*.json`.
- Board screen structure reflects target form direction (left menu + right dense row list).
