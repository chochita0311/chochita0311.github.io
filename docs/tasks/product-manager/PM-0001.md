# PM-0001: Board MVP Execution Plan

## Purpose
Define and execute the first working board slice for the web app:
1. One left tab board (`My Notes`)
2. Post list/search/detail flow (read-focused)
3. Local separated table artifacts in `docs/tables/localdb/`
4. Cafe-board style layout baseline (left menu + right table-like list)

## Standard Reference
- Follow `docs/tasks/product-manager/PM-OPERATING-STANDARD.md` as the reusable PM workflow baseline.
- PM-0001 is treated as the first applied reference case for this standard.

## In Scope
- Board foundation only (`mst_board`, `mst_post` local read)
- Static FE runtime with local JSON table files
- Search/list/detail UX for posts
- Board-focused DE/FE/QA delegation and verification

## Out of Scope
- Backend implementation (Kotlin/Spring)
- Post DB refinement/migration strategy
- Posting (create/edit/delete) feature implementation
- Auth/member/admin
- Comment/reply implementation
- Production deployment and operations

## Board Development Process (Step-by-Step)
1. Data baseline setup
- Create local table files under `docs/tables/localdb/`
- Prepare at least:
  - `mst_board.json`
  - `mst_post.json`
  - `log_sync_run.json` (temporary)
- Source data is imported from `English - Langs Studio`

2. FE read path stabilization
- FE reads board/posts only from `docs/tables/localdb/`
- FE list and detail must work without backend
- Search/filter/pagination should run on local cache

3. UX handoff alignment
- DE delivers board-specific UX guidance for:
  - left sidebar board information hierarchy
  - right-side table-like list hierarchy (title/author/date/views shape)
  - search/list/detail states
  - mobile + desktop behavior

4. QA verification alignment
- QA validates board baseline behaviors:
  - localdb loading and fallback state messages
  - list/search/detail functional correctness
  - desktop/mobile layout breakpoints and scroll behavior
  - edge/error scenarios (empty rows, missing fields, malformed dates)

5. Acceptance check
- Verify list/search/detail works with separated local table files
- Verify URL state and back-navigation behavior
- Verify basic board form resembles target structure:
  - left menu area
  - post list rows in table-like information density
- Record blocking points/materials for PM-0002 and PM-0003+

## Delegation

### Delegate to DE (`docs/tasks/designer/DE-0001.md`)
- Refine board UX for one-tab information architecture
- Specify state behavior for:
  - default/loading/empty/error/success
- Provide FE-ready handoff for mobile and desktop
- Deliver table-like list visual spec based on target board form

### Delegate to FE (`docs/tasks/frontend-engineer/FE-0001.md`)
- Implement localdb adapter using `docs/tables/localdb/*.json`
- Remove `notes.json` dependency from runtime
- Keep code path operational in local mode now
- Prepare API contract draft as PM-0003 material only
- Implement board page structure to match target form direction (left menu + dense row list)

### Delegate to QA (`docs/tasks/quality-assurance/QA-0001.md`)
- Build and execute PM-0001 board verification checklist
- Validate FE behavior on localdb mode for both desktop and mobile view
- Report reproducible defects and release risks for PM decision

## Deliverables
1. PM process definition for board development
2. DE and FE delegated task details
3. QA delegated verification checklist and result channel
4. Working FE app from `docs/tables/localdb/`
5. PM-0003 handoff materials list

## Acceptance Criteria
- PM process steps are explicit and executable.
- DE/FE/QA delegation is documented and actionable.
- FE works using separated local table files (`docs/tables/localdb/`).
- QA checklist is executable with clear pass/fail criteria.
- PM-0003 material list is captured.
